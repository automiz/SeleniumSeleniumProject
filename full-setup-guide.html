Selenium & REST Assured Enterprise Framework Blueprint

This master document serves as the end-to-end design layout, compilation workflow, and CLI execution guide for a hybrid UI/API test automation architecture optimized for enterprise core banking simulations. Built entirely with clean architectural patterns and command-line execution mapping.

1. Framework Directory Layout & Structural Upgrades

The application code, testing suites, utilities, and configuration profiles are completely isolated under the test resource trees to maintain a professional, production-ready framework state:

2. Maven Project Settings & Dependency Architecture

The framework relies on standard Maven compilation rules. The core build dependencies added to the pom.xml profile are optimized for testing both visual banking interfaces and background ledgers:

  • TestNG Engine: Drives the core testing execution flow, manages test annotations, and coordinates data pipeline setups.
  • Selenium Java Core: Provides standard browser controls, explicit synchronizations, and web page interactions.
  • REST Assured Core: Built to parse, send, and validate backend API banking transactions.
  • Apache POI (XSSF Models): Handles high-speed, direct memory access to open and read modern .xlsx data sheets.
  • Maven Surefire Plugin: Binds the Maven runtime test goals directly to execution suite configuration files.

3. Automated Data-Driven Strategy (Excel Layer)

To eliminate hardcoded constants from scripts, test parameters are completely separated into an external spreadsheet (./TestData.xlsx) under a specific data tab named Onboarding.

A. The Data Extraction Utility

The framework utilizes a dedicated extraction class that reads rows dynamically through the following structural process:

  • Opens a secure data pipeline to the file location using a Java FileInputStream.
  • Loads the entire workbook structure into an XSSFWorkbook tracker.
  • Navigates to the targeted data sheet tab using an XSSFSheet pointer.
  • Calculates the precise horizontal and vertical boundaries of the test matrix using the row and cell calculation methods.
  • Converts the cells into simple, clean text strings using a DataFormatter utility to prevent formatting errors.
  • Implements automated safety guards to check if an active row is empty, allowing the engine to gracefully skip empty rows instead of throwing a runtime error.

B. The Test Pipeline Injection

The data grid is packaged into a standard 2D Object grid array. A TestNG @DataProvider intercepts this grid and sequentially passes the values into the @Test methods as matching functional parameters, running the tests automatically for every data row.

4. Test Isolation Design (Inheritance & Base Controllers)

To follow clean coding practices, browser controls are split into two layers:

  • The Parent Controller (BaseTest): Holds the centralized setup and teardown rules. It manages browser instances, window sizes, and safely shuts down active session windows after a test runs.
  • The Test Case Suite (BankingTest): Inherits session variables from the parent class. It contains only the actual banking workflow logic, test verifications, and assertions.

5. Real-Time Interception Layer (TestNG Listeners)

A central test observer class implements the core ITestListener interface. It runs silently in the background of the execution suite to monitor testing states in real time:

  • Suite/Test Start: Intercepts execution initialization to generate clean console logs.
  • Failure Interception: The exact millisecond a validation checkpoint fails, the engine intercepts the active class driver instance, triggers a visual screenshot capture, formats it with a timestamp, and saves it inside an isolated folder path before the browser closes.
